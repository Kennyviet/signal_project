## Subsystem 1 – Alert Generation System
The Alert Generation System is designed around a single core responsibility: evaluate incoming patient vitals and trigger appropriate alerts in real time. The central class, AlertGenerator, acts as the decision engine. There is an AlertThreshold object that holds patient-specific rules, for example each patient may have an alert rule if their heart rate exceeds 130 bpm, which maintain the threshold logic separately from the evaluation logic. This allows for adding a new patient alert rule by simply updating their associated AlertThreshold object without needing to change the logic of the generator itself.

An Alert is simply a data object and consists of a patient’s identification number, what condition caused an alert to be triggered, what severity level of alert has been triggered and when the alert occurred. Keeping the Alert as an immutable, lightweight data object ensures that it may be logged, queued, and transmitted without having any state change due to the process of logging, queuing, or transmitting it. AlertManager is responsible for dispatching alerts to the right medical staff and logging them to DataStorage. This separation of dispatch from detection means the generator never needs to know who receives the alert or how.

Multiplicity is used deliberately: one AlertGenerator can reference multiple AlertThreshold objects (one per metric type, e.g., heart rate and blood pressure separately), and AlertManager aggregates zero or more alerts over time for history retrieval. Visibility choices reflect privacy requirements — patient data fields are private (-), and only necessary getters are public (+). Design is adaptable: Any new alert strategies can be created as additional subclasses of the AlertThreshold type without changing anything in the generator. This is similar to what was described in the assignment as the Strategy Pattern concept. Additionally, this allows for safety-critical requirements of the alert logic to be made to meet safety-critical requirements such as being auditable, traceable and easily modified per patient, and also meeting the need for being easily modified.

## subsystem 2 - Data Storage System
The Data Storage System uses controlled access methods to manage all patient vital information which travels through its single DataStorage interface that prevents any system component from accessing records. The interface design limits external systems to four operations which allow them to store data retrieve data delete data and purge data from the system. The system interface design enables users to change storage technology between in-memory storage and file-based storage and database storage without requiring any system modifications.

PatientData is a time-stamped, versioned object containing just one set of vitals. This versioning is by design: if there’s a change to the readings from the data source, then a new version of the same record will be recorded. The expiration of PatientData objects is handled by retentionDays, which is a property of DataStorage, through the method purgeExpired().

DataRetriever encapsulates all query logic used by medical staff. Rather than giving staff direct access to DataStorage, queries pass through DataRetriever, which first checks access permissions via hasAccess() before executing. Every successful query is recorded by AccessLog, which creates an audit trail linking the requester, patient, and timestamp. This satisfies the privacy and access control requirements: sensitive patient records are never exposed without both authorization and a corresponding log entry. Visibility modifiers reinforce this like storage internals (encryptionKey, records) are private, while retrieval methods are public only on DataRetriever, not on DataStorage directly. The overall design prioritizes accountability, traceability, and clear separation between who stores data and who is allowed to read it.

## Subsystem 3 - Patient Indentification System 
The Patient Identification System addresses one of the most privacy-sensitive responsibilities in the CHMS: linking raw simulator data to real hospital records. The design deliberately restricts direct access to HospitalPatient objects — they are only reachable through PatientIdentifier, which performs matching logic and assigns a confidence score before returning a result. If confidence falls below a threshold, no match is returned; instead, a MismatchRecord is created and escalated.

IdentityManager sits at the top of this subsystem and acts as the single point of coordination. It maintains a list of active PatientIdentifier objects (one can exist per data stream or data source), holds the mismatch log, and provides an integrity report for auditors. This centralization means that even if multiple streams are running simultaneously, all identification decisions are tracked in one place.

The HospitalPatient table is designed to contain just enough information for the identification of a patient and monitoring them such as hospital ID, patient name, date of birth, ward, and medical history. The HospitalPatient table intentionally does not contain any authentication credentials or financial data; this is done to maintain the principle of least exposure. All fields are private, accessible only through getters. MismatchRecord serves a dual purpose: it logs anomalies for auditing and it triggers an escalation path (escalate()) so that unresolved mismatches do not silently drop data. This is critical in a safety setting — a missed patient link could mean vitals go unmonitored. The multiplicity choices reflect real-world constraints: one IdentityManager oversees one-to-many PatientIdentifier objects, each of which maps to at most one HospitalPatient at a time, keeping matching unambiguous and auditable.

## Subsystem 4 - Data Access Layer 
The Data Access Layer acts as the boundary between external data sources (such as the signal generator) and the rest of the CHMS. The system uses an abstract base class called DataListener to define common input source interfaces through its methods start(), stop(), onDataReceived(), and its abstract method listen(). The three concrete implementations of the system, TCPDataListener and WebSocketDataListener and FileDataListener, support their own implementation of listen() according to their respective protocols while they share common lifecycle behavior. This abstraction ensures the rest of the system depends only on DataListener, remaining independent of how data is received.

DataParser is intentionally separated from listeners to maintain clear responsibility boundaries. While listeners handle raw data input, the parser interprets and structures it. This separation allows flexibility, such as switching between JSON and CSV formats without modifying listener logic. Methods like detectFormat() and validate() ensure incoming data is correctly identified and structurally valid before further processing.

Finally, DataSourceAdapter manages the transition to storage. It receives validated PatientData objects, temporarily buffers them in a queue, and writes them to DataStorage in controlled batches. This buffering prevents overload during high data bursts. The system requires complete isolation because listener subclasses need to keep their connection details confidential. The system allows new data sources to be added through Bluetooth device integration because it only requires the creation of a new listener.
